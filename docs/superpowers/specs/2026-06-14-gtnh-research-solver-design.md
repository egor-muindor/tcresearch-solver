# GTNH Research Solver — Спецификация

**Дата:** 2026-06-14
**Статус:** одобрено пользователем; прошло 3 раунда ревью Codex (замечания учтены — см. Приложения A, B, C)
**Ветка:** `feature/gtnh-research-solver`

## 0. Контекст и цель

Существующий репозиторий (`Thaumcraft 4.x Research Helper`) умеет искать минимальный путь
между **двумя** аспектами в дереве комбинаций. Берём из него **только данные** о переходах
и крафтах аспектов (`js/version_dictionary.js`, `js/addon_dictionary.js`,
`js/translation_dictionary.js`) и **полностью переписываем** алгоритм и интерфейс.

Цель — полноценный решатель исследований GTNH на гекс-доске радиуса R∈[2,5], который:

1. Заполняет пустые гексы аспектами, **соединяя все якоря** (механика TC4/GTNH).
2. **Обходит dead hex** (отключённые клетки).
3. Учитывает **инвентарь пользователя**: использует в первую очередь аспекты, которых много,
   **с учётом крафта** новых аспектов из имеющихся; дефицитные (ниже порога, по умолчанию 50)
   — в последнюю очередь.
4. Сохраняет инвентарь и состояние доски в браузере (localStorage).
5. Умеет **вычесть** предложенные/потраченные аспекты из инвентаря.
6. Позволяет **ручное редактирование** доски, **валидацию** решения и **продолжение**
   решения, начатого пользователем.
7. Работает **целиком в браузере**, без зависания UI, с отображением прогресса и возможностью отмены.

Старые файлы (`js/`, `index.html`, `css/`, …) **не трогаем** — новое приложение пишем рядом.

## 1. Зафиксированные решения

| Вопрос | Решение |

| Что решаем | **Якоря + соединить все** (Steiner-подобная задача на гексах) |
| Источник данных аспектов | **Данные из репо**: TC 4.2.2.0 base + addons (Gregtech, Forbidden Magic, Magic Bees) — нормативны для v1 |

## 2. Терминология и модель данных

### 2.1 Аспекты и граф

- **Аспект** — узел с ключом (англ., напр. `air`, `void`) и латинским отображением (`translate`,
  напр. `aer`, `vacuos`) + иконка `aspects/color/<latin>.png`.
- **Праймалы** (6): `air`, `earth`, `fire`, `water`, `order`, `entropy` — не имеют компонентов.
- **Компаунд** — `combinations[X] = [c1, c2]`: аспект X крафтится из компонентов c1 и c2.
- **Граф G:** неориентированное ребро X–Y ⇔ один аспект является **прямым компонентом** другого
  (X∈combinations[Y] или Y∈combinations[X]). Петель нет; «братья» (общий родитель/ребёнок) **не**
  связаны. Это ровно граф `connect(compound, component)` из старого кода. Смежность хранится как
  дедуплицированный `Set` на аспект.
- **Связь валидна** между аспектами A и B ⇔ A≠B и существует ребро A–B в G.

**Универсум аспектов** определяется явно как объединение: праймалы ∪ ключи `combinations` ∪
**компоненты** всех рецептов ∪ addon-аспекты. Для **каждого** члена универсума обязательны метаданные
(`translate` и иконка) — отсутствие проверяется на старте.

**Стартовая валидация данных (fail loudly):** при инициализации проверяются — отсутствие циклов и
self-reference в рецептах, отсутствие дублей рёбер, наличие всех компонентов в универсуме, наличие
`translate`/иконки для каждого аспекта. Любое нарушение → исключение с именем аспекта (не молчаливый
пропуск, не зацикливание). Эталонный фикстур фиксирует точное множество рёбер графа.

**Фиделити к GTNH:** данные репо — это generic Thaumcraft 4.2.2 + Gregtech-addon; полное соответствие
именно сборке GTNH **не гарантировано**. По решению пользователя рецепты репо **нормативны для v1**.
Архитектура данных (`src/data/aspects.ts`) допускает замену набора без изменений ядра (см. §10).

### 2.2 Праймал-разложение

`primalVec(X): Map<primal, number>` — мультимножество праймалов, из которых в итоге крафтится X:
- праймал p → `{p: 1}`;
- компаунд X=[c1,c2] → `primalVec(c1) + primalVec(c2)` (поэлементная сумма).

Мемоизируется. Граф комбинаций — DAG от праймалов вверх; **глобальная проверка отсутствия циклов**
выполняется один раз до любых вычислений стоимости/аллокации (общая с §2.1).

### 2.3 Гекс-доска

- Координаты — cube/axial. Доска радиуса R: все клетки с расстоянием от центра ≤ R.
  Кол-во клеток: R2=19, R3=37, R4=61, R5=91.
- Соседи — 6 направлений.
- **Единая модель состояния клетки** (взаимоисключающие варианты):
  `DEAD` | `ANCHOR(aspect)` | `EMPTY` | `PLACED(aspect, locked)`.
  - `ANCHOR` — предзадан пользователем как требуемый аспект исследования; неизменяем solver-ом;
    **не списывается** из инвентаря (даётся research note).
  - `PLACED, locked=true` — ручное размещение пользователя (фиксируется при «Continue Solve»).
  - `PLACED, locked=false` — размещение, найденное solver-ом (только эти клетки списываются, §4.3).
- **Нет** отдельной dead-маски: «мёртвость» — это состояние клетки. `PLACED_ON_DEAD` — только
  диагностика при разборе некорректной/внешней сериализации.
- Сериализация доски — компактная: `{schemaVersion, radius, cells:[{coord, state, aspect?, locked?}]}`
  (хранятся только не-`EMPTY` клетки). При загрузке — валидация радиуса, координат, id аспектов,
  числовых полей; некорректная сериализация → понятная ошибка/сброс к дефолту, без падения.

## 3. Правила валидного решения и валидация входа

Решение — присвоение аспектов подмножеству клеток в состоянии `EMPTY` (часть `EMPTY` может остаться
пустой), при котором:

1. **Якоря и locked-клетки сохраняют свои аспекты.**
2. **Полная валидность доски:** для **любой** пары соседних *заполненных* клеток (anchor/placed)
   аспекты **разные** и связаны ребром в G. (Включая случайные касания разных веток — они тоже
   должны быть валидны; поэтому оптимальные решения — тонкие непересекающиеся цепочки.)
3. **Dead hex** остаются пустыми (solver их не использует и обходит).
4. **Connect-all:** все клетки-якоря лежат в **одной** связной компоненте заполненных клеток
   (связность по соседству на доске). `locked`-клетки — неизменяемые ограничения (учитываются в
   полной валидности §3.2), но **не обязаны** соединяться с якорями, если сами не являются якорями
   (они могут быть использованы цепочкой, но изолированный locked-остров валиден).

`validate(board)` → `{ valid: boolean, errors: Array<{type, cells}> }`, где
`type ∈ {INVALID_LINK, SAME_ASPECT_ADJACENT, ANCHORS_DISCONNECTED, PLACED_ON_DEAD, MALFORMED}`.

**Предвалидация входа (обязательна перед поиском):** доска со стартовыми `ANCHOR`+locked прогоняется
через `validate()`. Если стартовое состояние уже невалидно (напр. соседние якоря несвязаны, что
решателю не починить) — поиск **не запускается**, возвращается статус `INVALID_INPUT` с указанием
клеток. Это устраняет случай «нерешаемо из-за входа», который ядро не может исправить.

## 4. Модель инвентаря (стоимость, выполнимость, аллокация)

### 4.1 Параметры и ограничения

- `supply: Map<aspect, number>` — введённые пользователем количества. Требования: **неотрицательные
  целые**; иначе вход отклоняется с сообщением.
- `threshold` (по умолчанию **50**, **строго > 0**) — граница «мало». `supply[X] ≥ threshold` ⇒
  X «изобильный».
- Константы: `base > 0`, `k ≥ 0` (неотрицательны ⇒ отсутствуют отрицательные стоимости — корректность
  Dijkstra/A*). Число клеток входит во вторую компоненту цели как **литеральный счётчик** (отдельной
  `cellCost` нет). `ε` из модели **исключён** (предпочтение мелких крафтов выражается точной
  аллокацией §4.3, а не магической добавкой к стоимости).

### 4.2 Скарсити-проксии и цель оптимизации

**Цель — лексикографическая пара `(Σ scarcity, число solver-клеток)`.** `craftOps` — отчётная
метрика (из аллокации §4.3), в порядке поиска НЕ участвует.

**Истинная стоимость дефицита доски** определяется только §4.3 (точная аллокация по запасам, с учётом
конкуренции за общие компоненты). Для **управления поиском и отсечения** используется покомпонентная
нижняя оценка дефицита — `obtainCost`, считаемая **независимо на аспект** (без учёта конкуренции):

```
obtainCost(X) = min( directPenalty(X),  obtainCost(c1) + obtainCost(c2) )   // без ε
directPenalty(X):
   supply[X] ≥ threshold     → 0
   0 < supply[X] < threshold → base + k·(threshold - supply[X])
   supply[X] == 0            → +∞                      // напрямую нельзя, только крафт
```

- Праймалы не крафтятся ⇒ их `obtainCost = directPenalty`.
- `obtainCost` считается DP по DAG на стартовом инвентаре, мемоизируется.
- **Ключевое свойство (для допустимости, см. §5.2):** независимая (без конкуренции) сумма
  `Σ_{placed} obtainCost(aspect)` **≤** истинного совместного дефицита аллокации (§4.3), т.к.
  независимая оптимизация игнорирует конкуренцию за общие запасы и потому не дороже совместной. Значит
  это валидная **нижняя** оценка истинной стоимости дефицита — её можно использовать в `g` и `h`.
- `globalMinObtain = min_{X∈универсум} obtainCost(X)` — используется как вес клетки в релаксации §5.2.

### 4.3 Точная аллокация: истинная стоимость, выполнимость, списание

Для **полной доски-кандидата** вычисляется точная аллокация ресурсов по запасам:

- **Задача аллокации (MRP-структура):** дано мультимножество размещённых solver-аспектов
  `demand[X]`. Каждую требуемую единицу X удовлетворяем либо из `supply[X]`, либо крафтом X из 1 ед.
  c1 и 1 ед. c2 (рекурсивно, что добавляет спрос на компоненты). Сломать компаунд нельзя.
- **Формальная постановка (ILP).** Переменные на аспект X: `direct[X] ≥ 0` (взято из запаса),
  `craft[X] ≥ 0` (скрафчено), целые; `craft[X]=0` для праймалов. `mult(X,Y)` = кратность X в рецепте
  Y (учитывает повторы, напр. `stone=earth+earth` ⇒ `mult(earth,stone)=2`).
  - Баланс: `direct[X] + craft[X] = placedDemand[X] + Σ_Y mult(X,Y)·craft[Y]`
    (произведённое = поставлено на доску + израсходовано как компонент скрафченных родителей).
  - Запас: `direct[X] ≤ supply[X]`.
  - **Целевая:** `scarcityCost = Σ_X direct[X]·directPenalty(X)` (штраф несут только прямые изъятия из
    запаса; крафт «бесплатен», но тянет прямые изъятия компонентов). `craftOps = Σ_X craft[X]`.
  - `directPenalty` — **фиксированный per-unit** штраф из §4.2 (зависит от *стартового* `supply[X]`, не
    от глубины изъятия) ⇒ `scarcityCost` линейна; это и обеспечивает свойство нижней оценки §4.2.
- **Метод — ТОЧНЫЙ и порядко-независимый:** настоящий целочисленный решатель (branch-and-bound ILP)
  **или** исчерпывающая мемоизированная свёртка по DAG. **Утверждение об интегральности LP снято** —
  для общей binary-BOM она неверна (контрпример: `X=A+B, Y=A+C, Z=B+C`, спрос по 1, `supply A=B=C=1`;
  дробное 0.5 крафтит 1.5, целочисленно — максимум 1), а задача в худшем случае NP-трудна
  (многомерная упаковка). **Запрещены** жадность и любые отсечения, способные вернуть ложное
  «infeasible». Инстансы v1 крошечные (мало различных размещённых аспектов, малая глубина крафта) ⇒
  исчерпывающая мемоизация практична; задаётся **явный бюджет аллокатора** (узлы/время); при
  (патологическом) исчерпании бюджета кандидат помечается как *не-доказанный* (не `feasible`, но и не
  `infeasible`) и не используется для статусов `OPTIMAL`/`INFEASIBLE_INVENTORY`.
  Результат: `{ feasible: bool|unknown, scarcityCost, craftOps, leafConsumption: Map<aspect, number> }`,
  где `leafConsumption[X] = direct[X]` — реально списываемые прямые расходы из `supply`.
- **Истинная стоимость доски** = `(scarcityCost, число solver-клеток)` (лексикографически).
  `feasible=false` ⇒ доска не может быть обеспечена запасами (см. инкумбент в §5.2).
- **Что списывается.** Заводится baseline-ревизия доски *до* Auto Solve (якоря + ручные locked).
  В `demand` идут **только новые** клетки `PLACED, locked=false`. Якоря и ранее введённые вручную
  аспекты **не** списываются.
- **Subtract used.** Кнопка вычитает `leafConsumption` из инвентаря; защита от ухода в минус; UI
  показывает разбивку «прямо потрачено / скрафчено» (`craftOps`).

Тестируется отдельно: кейсы, где жадность ложно отвергает выполнимую аллокацию (напр. `light` раньше
двух `aura`, конкуренция за `air`), — точный метод обязан находить выполнимую аллокацию.

## 5. Алгоритм solver-а

### 5.1 Постановка и границы

Node-weighted Steiner tree на произведении «доска × метки-аспекты» с ограничением полной валидности.
NP-трудно в общем случае; доски малы (≤91 клетка). Поддерживается **2–8 якорей** (тестируемый максимум);
поведение при **0/1 якоре** — тривиально «уже решено» (пустой набор размещений). Сверх максимума — UI
предупреждает и не запускает поиск.

### 5.2 Ядро: cost-bounded branch-and-bound (A*/IDA*) с точной выборкой инкумбента

- Состояние — частичное множество размещённых клеток (`coord → aspect`), стартующее от якорей +
  locked-клеток. Расширение: добавить аспект в пустую клетку, соседнюю с текущей структурой; при
  каждом размещении — проверка валидности против **всех** уже стоящих соседей ⇒ полная валидность и
  обход dead hex.
- **Единый лексикографический компаратор** для всех сравнений и отсечений (никаких «покомпонентно
  независимых» отсечений). Пара = `(scarcity, cells)`.
- **Нижняя оценка f = g_lb ⊕ h:**
  - `g_lb` = `(Σ_{placed} obtainCost, cellsPlaced)` — нижняя оценка истинной стоимости поставленного
    (§4.2), плюс точное число клеток.
  - `h` = **оптимум релаксации на ОСТАТКЕ** (cost-to-go), считаемый на *текущем состоянии*:
    1. связные компоненты заполненных клеток, **содержащие якоря**, стягиваются в **обязательные
       терминалы**; компоненты только из `locked`-клеток (без якоря) — **опциональные** вершины
       Штейнера (их соединять не требуется, см. §3.4), а не терминалы;
    2. уже поставленные клетки имеют **вес 0** (стянуты/бесплатны);
    3. вес каждой свободной клетки = `globalMinObtain` (§4.2) — допустимый недооценочный вес
       (≤ стоимости любого реально поставленного туда аспекта; множество достижимых меток = все
       аспекты);
    4. на этом релаксированном графе **клеток** (без меток и без полной валидности) решается задача
       Штейнера **точно** алгоритмом **Dreyfus–Wagner** за O(3^k·n + 2^k·n²), k = число терминалов ≤8.
    - Релаксация (снятие меток/валидности + стягивание оплаченного) ⇒ нижняя оценка остатка ⇒
      **допустимость**. Представляется вектором `(hScarcity, hCells)`: `hScarcity` — стоимость
      релаксации, `hCells` — число внутренних клеток мин-клеточного Штейнера (отдельная Dreyfus–Wagner
      с единичными весами; нижняя оценка остатка по клеткам). Лексикографически допустимо, т.к. каждый
      компонент h — нижняя оценка соответствующего остатка. `h=(hScarcity,0)` — безопасный дефолт;
      `(hScarcity,hCells)` — доказанно допустимое усиление.
    - **MST по попарным расстояниям ОТВЕРГНУТ** (переоценивает для общего центра вроде `air`,
      к которому примыкают `tree/aura/light`: MST даёт ~2w вместо w) — неприемлем, не используется.
- **Инкумбент — дешевейшая ВЫПОЛНИМАЯ полная доска.** При достижении полной валидной доски считается
  её **истинная стоимость и выполнимость** (§4.3):
  - `feasible=true` и стоимость < инкумбента ⇒ обновить инкумбент;
  - `feasible=false` ⇒ **отбросить** (никогда не становится инкумбентом, не задаёт верхнюю границу);
  - `feasible=unknown` (аллокатор исчерпал бюджет §4.3) ⇒ кандидат **не** обновляет инкумбент и **не**
    отвергается окончательно; взводится флаг `anyUnknownCompetitive`, если его нижняя оценка стоимости
    могла бы побить инкумбент (т.е. кандидат потенциально конкурентен). Такой кандидат блокирует
    доказательные статусы (см. §5.4).
- **Инварианты корректности (обязательны):**
  1. аллокация §4.3 **точная** (без ложных «infeasible»);
  2. невыполнимые полные доски **не** влияют на инкумбент/границу;
  3. поиск **продолжается за каждую достигнутую цель** (не останавливается на первой полной доске);
  4. отсечение частичной ветки — только по `f` ⪰ инкумбента единым компаратором; частичные доски **не**
     отсекаются по выполнимости (выполнимость проверяется лишь на полных досках);
  5. **оптимум доказан**, когда минимальный `f` фронтира ⪰ инкумбента (исчерпывающе);
  6. **bound-pruning по инкумбенту включается ТОЛЬКО после появления конечного выполнимого инкумбента**
     (до этого поиск идёт без отсечения по границе, в пределах бюджета). Иначе сентинел-инкумбент
     `(+∞,+∞)` мог бы отсечь link-валидные, но невыполнимые по запасам доски до их обнаружения и
     испортить различение `UNSAT_PROVEN`/`INFEASIBLE_INVENTORY`. (Безопасно отсекать ветки с
     `g_lb.scarcity = +∞` — недостижимые ни одной аллокацией — это не влияет на статус.)
  7. флаг `anyValidBoardFound` фиксирует существование хотя бы одной link-валидной полной доски
     (независимо от выполнимости): исчерпывающий поиск без таких досок ⇒ `UNSAT_PROVEN`; есть валидные,
     но ни одной выполнимой ⇒ `INFEASIBLE_INVENTORY`.
- **Anytime:** сначала быстрое допустимое решение (посев §5.3) для раннего инкумбента, затем улучшение
  до оптимума или до бюджета.

### 5.3 Посев (подход B): попарный Dijkstra + сшивка — недоверенные кандидаты

Для пар якорей — дешёвые пути в произведении-графе `(cell, aspect)`, метрическое замыкание, сшивка.
**Посев недоверенный:** Dijkstra сам по себе не гарантирует полную валидность (инцидентные касания,
совпадающие координаты с разными метками, примыкание третьего якоря, конфликты слияния). Поэтому посев
— **набор кандидатов**: слияние → полная `validate()` → при конфликте отвергается/ремонтируется
бэктрекингом. В инкумбент кандидат попадает только после `validate()` **и** проверки выполнимости §4.3.

### 5.4 Статусы результата и фолбэк

Результат solver-а имеет явный статус:

| Статус | Когда |
|---|---|
| `OPTIMAL` | исчерпывающий поиск завершён, найден выполнимый инкумбент, оптимум доказан |
| `FEASIBLE_TIMEOUT` | есть выполнимый инкумбент, но оптимальность не доказана (бюджет/beam) |
| `UNKNOWN_TIMEOUT` | бюджет/beam исчерпан, выполнимое решение **не найдено** и невыполнимость/UNSAT **не доказаны** |
| `INFEASIBLE_INVENTORY` | **исчерпывающим** точным поиском доказано: валидные доски есть, но ни одна не обеспечивается запасами |
| `UNSAT_PROVEN` | **исчерпывающим** поиском доказано: валидной доски не существует |
| `CANCELLED` | прервано пользователем |
| `INVALID_INPUT` | стартовый вход невалиден (§3) |

- `OPTIMAL`, `INFEASIBLE_INVENTORY`, `UNSAT_PROVEN` выдаются **только** из исчерпывающего точного поиска.
- **Вердикт аллокатора `unknown` (§4.3/§5.2) блокирует доказательные статусы.** Если `anyUnknownCompetitive`
  взведён (есть потенциально конкурентный кандидат с неразрешённой выполнимостью), результат **не может**
  быть `OPTIMAL` или `INFEASIBLE_INVENTORY`; он деградирует до `FEASIBLE_TIMEOUT` (если выполнимый
  инкумбент уже есть) либо `UNKNOWN_TIMEOUT` (если инкумбента нет). `UNSAT_PROVEN` не затрагивается
  (зависит только от `anyValidBoardFound`, а не от аллокатора).
- **Beam-фолбэк** на тяжёлых досках даёт `FEASIBLE_TIMEOUT` (если найден инкумбент) или
  `UNKNOWN_TIMEOUT` — **никогда** `*_PROVEN`/`INFEASIBLE_INVENTORY`. Любое усечение **явно логируется**.

### 5.5 Производительность, отмена, бюджеты

- Предвычисления: смежность клеток, битмаски доски, кэш эвристики, разложение аспектов, `obtainCost`.
- Порядок ветвления: сначала дешёвые/перспективные аспекты; отсечение — единым компаратором (§5.2).
- **Бюджеты по радиусам** (узлы/время/память) задаются явно для R2–R5 и проверяются бенчами (§7).
- **Web Worker.** Поиск идёт **чанками** с возвратом управления в асинхронный планировщик между
  чанками (проверка флага отмены/сообщений). **Гарантированная отмена — hard `worker.terminate()`**
  с пересозданием воркера. Прогресс через `postMessage` (узлы / лучшая стоимость / время / статус).
- Решатель — **чистый TS без DOM** (тестируется напрямую `bun test` и используется воркером).

## 7. Тесты и производительность (test-first для ядра, ДО UI)

Порядок: сначала проходит тесты алгоритм, потом пишем UI. **Типизация:** `bunx tsc --noEmit` —
обязательный шаг (Vite/bun только транспилируют, не проверяют типы).

`bun test` покрывает:
- **hex.ts**: соседи, расстояния, генерация доски по радиусам, сериализация/десериализация.
- **aspectGraph.ts**: смежность из данных (эталонный edge-set фикстур), праймал-разложение, защита от
  циклов (fail loudly), `isValidLink` (запрет одинаковых соседей и «братьев»).
- **board.ts**: полная валидность (INVALID_LINK, SAME_ASPECT_ADJACENT, инцидентные касания,
  PLACED_ON_DEAD), связность якорей (ANCHORS_DISCONNECTED), isComplete; (де)сериализация с миграцией.
- **inventory.ts**: directPenalty по порогу; obtainCost с крафтом (изобильный компонент «спасает»
  дефицитный компаунд) и его монотонность; свойство нижней оценки `Σ obtainCost ≤ истинной аллокации`;
  **точная аллокация §4.3** (порядко-независимость; находит выполнимую аллокацию там, где жадность
  падает; корректные `leafConsumption`/`craftOps`); отказ на отрицательных/нецелых входах.
- **solver.ts**: найденное решение **всегда** валидно и связно (инвариант); обход dead hex; мульти-
  якоря (до 8); **предвалидация входа** → `INVALID_INPUT`; **допустимость эвристики при частичном
  состоянии** (h ≤ фактической стоимости-остатка на наборе кейсов, в т.ч. со стянутыми компонентами и
  нулевым весом оплаченных клеток); **B&B выбирает дешевейшую ВЫПОЛНИМУЮ доску** (кейс, где самая
  дешёвая по связям доска невыполнима по запасам, а оптимум — более дорогая выполнимая); статусы:
  `UNSAT_PROVEN`/`INFEASIBLE_INVENTORY`/`OPTIMAL` только из исчерпывающего поиска, `UNKNOWN_TIMEOUT`
  при усечении без инкумбента; предпочтение изобильных аспектов; оптимальность на малых кейсах;
  **locked-острова** не делаются обязательными терминалами (изолированный locked не раздувает `h` и
  не блокирует оптимум); **различение `UNSAT_PROVEN` vs `INFEASIBLE_INVENTORY`** (валидная доска есть,
  но запасов нет ⇒ INFEASIBLE; валидной доски нет ⇒ UNSAT) и отсутствие bound-pruning до выполнимого
  инкумбента; **бюджет аллокатора** (патологический инстанс ⇒ `unknown`, а не ложный вердикт).
- **Бенчи**: время/узлы по R2–R5 при типовых наборах якорей; фиксируем бюджеты §5.5 и применяем
  оптимизации. Бенч — отдельный скрипт, не блокирующий обычный прогон тестов.

## 8. Интерфейс (по референс-скриншоту)

- **Шапка:** заголовок; селектор радиуса (2–5); инструменты **Dead Hex**, **Erase**, **Clear**,
  **Auto Solve**; новые **Validate** и **Continue Solve**.
- **Лево — палитра `Aspects`:** клик по аспекту выбирает «кисть»; клик по гексу ставит якорь
  (или ручное размещение в режиме редактирования).
- **Центр — пергаментная гекс-доска (SVG):** иконки аспектов, dead hex заштрихованы; невалидные
  связи подсвечиваются красным; статус-строка снизу.
- **Право — `Inventory`:** список аспектов с редактируемыми количествами; поле порога «мало» (50);
  кнопка **Subtract used** + разбивка потраченного (прямо/крафт).
- **Прогресс:** во время решения — прогресс-бар (узлы / лучшая стоимость / время / статус) и
  кнопка **Cancel** (hard-terminate воркера).
- **Ручной режим:** клик по пустому гексу ставит выбранный аспект (locked); **Validate** проверяет
  полную валидность + связность и подсвечивает проблемы; **Continue Solve** фиксирует ручные
  размещения как ограничения и достраивает остальное.
- **Статусы решателя** отображаются человекочитаемо (Оптимально / По таймауту / Неизвестно (таймаут) /
  Не хватает запасов / Решения нет / Отменено / Некорректный ввод).
- **Атрибуция:** видимая ссылка CC-BY-4.0 на исходные данные/иконки (по README репо) + уведомление
  в репозитории.
- Всё состояние (инвентарь, порог, доска, dead hex) — в **localStorage** со `schemaVersion`.
