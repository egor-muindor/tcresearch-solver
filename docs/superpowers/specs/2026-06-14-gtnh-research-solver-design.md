# GTNH Research Solver — Спецификация

**Дата:** 2026-06-14
**Статус:** одобрено пользователем; прошло ревью Codex (замечания учтены — см. Приложение A)
**Ветка:** `feature/gtnh-research-solver`

## 0. Контекст и цель

Существующий репозиторий (`Thaumcraft 4.x Research Helper`) умеет искать минимальный путь
между **двумя** аспектами в дереве комбинаций. Берём из него **только данные** о переходах
и крафтах аспектов (`js/version_dictionary.js`, `js/addon_dictionary.js`,
`js/translation_dictionary.js`) и **полностью переписываем** алгоритм и интерфейс.

Цель — полноценный решатель исследований GTNH на гекс-доске радиуса R∈[2,5], который:

1. Заполняет пустые гексы аспектами, **соединяя все якоря** (механика TC4/GTNH).
2. **Обходит dead hex** (отключённые клетки).
3. Учитывает **инвентарь пользователя**: использует в первую очередь аспекты, которых много,
   **с учётом крафта** новых аспектов из имеющихся; дефицитные (ниже порога, по умолчанию 50)
   — в последнюю очередь.
4. Сохраняет инвентарь и состояние доски в браузере (localStorage).
5. Умеет **вычесть** предложенные/потраченные аспекты из инвентаря.
6. Позволяет **ручное редактирование** доски, **валидацию** решения и **продолжение**
   решения, начатого пользователем.
7. Работает **целиком в браузере**, без зависания UI, с отображением прогресса и возможностью отмены.

Старые файлы (`js/`, `index.html`, `css/`, …) **не трогаем** — новое приложение пишем рядом.

## 1. Зафиксированные решения

| Вопрос | Решение |
|---|---|
| Что решаем | **Якоря + соединить все** (Steiner-подобная задача на гексах) |
| Источник данных аспектов | **Данные из репо**: TC 4.2.2.0 base + addons (Gregtech, Forbidden Magic, Magic Bees) — нормативны для v1 |
| Модель инвентаря | **Скарсити-вес + рекурсивный крафт**, порог «мало» по умолчанию **50** |
| Стек | **Vite + TypeScript**, **bun** (зависимости/сборка/тесты), solver в **Web Worker** |
| Повторы (одинаковые соседние аспекты) | **Запрещены** — соседи должны быть *разными* связанными аспектами |
| Проверка связности | **Полная валидность доски** — каждая пара соседних заполненных гексов обязана быть валидной связью |
| Цель оптимизации | **Лексикографически** `(суммарный дефицит, число клеток, число крафт-операций)` |

## 2. Терминология и модель данных

### 2.1 Аспекты и граф

- **Аспект** — узел с ключом (англ., напр. `air`, `void`) и латинским отображением (`translate`,
  напр. `aer`, `vacuos`) + иконка `aspects/color/<latin>.png`.
- **Праймалы** (6): `air`, `earth`, `fire`, `water`, `order`, `entropy` — не имеют компонентов.
- **Компаунд** — `combinations[X] = [c1, c2]`: аспект X крафтится из компонентов c1 и c2.
- **Граф G:** неориентированное ребро X–Y ⇔ один аспект является **прямым компонентом** другого
  (X∈combinations[Y] или Y∈combinations[X]). Петель нет; «братья» (общий родитель/ребёнок) **не**
  связаны. Это ровно граф `connect(compound, component)` из старого кода. Смежность хранится как
  дедуплицированный `Set` на аспект.
- **Связь валидна** между аспектами A и B ⇔ A≠B и существует ребро A–B в G.

**Универсум аспектов** определяется явно как объединение: праймалы ∪ ключи `combinations` ∪
**компоненты** всех рецептов ∪ addon-аспекты. Для **каждого** члена универсума обязательны метаданные
(`translate` и иконка) — отсутствие проверяется на старте.

**Стартовая валидация данных (fail loudly):** при инициализации проверяются — отсутствие циклов и
self-reference в рецептах, отсутствие дублей рёбер, наличие всех компонентов в универсуме, наличие
`translate`/иконки для каждого аспекта. Любое нарушение → исключение с именем аспекта (не молчаливый
пропуск, не зацикливание). Эталонный фикстур фиксирует точное множество рёбер графа.

**Фиделити к GTNH:** данные репо — это generic Thaumcraft 4.2.2 + Gregtech-addon; полное соответствие
именно сборке GTNH **не гарантировано**. По решению пользователя рецепты репо **нормативны для v1**.
Архитектура данных (`src/data/aspects.ts`) допускает замену набора без изменений ядра (см. §10).

### 2.2 Праймал-разложение

`primalVec(X): Map<primal, number>` — мультимножество праймалов, из которых в итоге крафтится X:
- праймал p → `{p: 1}`;
- компаунд X=[c1,c2] → `primalVec(c1) + primalVec(c2)` (поэлементная сумма).

Мемоизируется. Граф комбинаций — DAG от праймалов вверх; **глобальная проверка отсутствия циклов**
выполняется один раз до любых вычислений стоимости/аллокации (общая с §2.1).

### 2.3 Гекс-доска

- Координаты — cube/axial. Доска радиуса R: все клетки с расстоянием от центра ≤ R.
  Кол-во клеток: R2=19, R3=37, R4=61, R5=91.
- Соседи — 6 направлений.
- **Единая модель состояния клетки** (взаимоисключающие варианты):
  `DEAD` | `ANCHOR(aspect)` | `EMPTY` | `PLACED(aspect, locked)`.
  - `ANCHOR` — предзадан пользователем как требуемый аспект исследования; неизменяем solver-ом;
    **не списывается** из инвентаря (даётся research note).
  - `PLACED, locked=true` — ручное размещение пользователя (фиксируется при «Continue Solve»).
  - `PLACED, locked=false` — размещение, найденное solver-ом (только эти клетки списываются, §4.3).
- **Нет** отдельной dead-маски: «мёртвость» — это состояние клетки. `PLACED_ON_DEAD` — только
  диагностика при разборе некорректной/внешней сериализации.
- Сериализация доски — компактная: `{schemaVersion, radius, cells:[{coord, state, aspect?, locked?}]}`
  (хранятся только не-`EMPTY` клетки). При загрузке — валидация радиуса, координат, id аспектов,
  числовых полей; некорректная сериализация → понятная ошибка/сброс к дефолту, без падения.

## 3. Правила валидного решения и валидация входа

Решение — присвоение аспектов подмножеству клеток в состоянии `EMPTY` (часть `EMPTY` может остаться
пустой), при котором:

1. **Якоря и locked-клетки сохраняют свои аспекты.**
2. **Полная валидность доски:** для **любой** пары соседних *заполненных* клеток (anchor/placed)
   аспекты **разные** и связаны ребром в G. (Включая случайные касания разных веток — они тоже
   должны быть валидны; поэтому оптимальные решения — тонкие непересекающиеся цепочки.)
3. **Dead hex** остаются пустыми (solver их не использует и обходит).
4. **Connect-all:** все клетки-якоря лежат в **одной** связной компоненте заполненных клеток
   (связность по соседству на доске).

`validate(board)` → `{ valid: boolean, errors: Array<{type, cells}> }`, где
`type ∈ {INVALID_LINK, SAME_ASPECT_ADJACENT, ANCHORS_DISCONNECTED, PLACED_ON_DEAD, MALFORMED}`.

**Предвалидация входа (обязательна перед поиском):** доска со стартовыми `ANCHOR`+locked прогоняется
через `validate()`. Если стартовое состояние уже невалидно (напр. соседние якоря несвязаны, что
решателю не починить) — поиск **не запускается**, возвращается статус `INVALID_INPUT` с указанием
клеток. Это устраняет случай «нерешаемо из-за входа», который ядро не может исправить.

## 4. Модель инвентаря (стоимость)

### 4.1 Параметры и ограничения

- `supply: Map<aspect, number>` — введённые пользователем количества. Требования: **неотрицательные
  целые**; иначе вход отклоняется с сообщением.
- `threshold` (по умолчанию **50**, **строго > 0**) — граница «мало». `supply[X] ≥ threshold` ⇒
  X «изобильный».
- Константы стоимости: `base > 0`, `k ≥ 0`, `ε > 0`, `cellCost > 0` (см. §4.2). Все неотрицательны ⇒
  отсутствуют отрицательные рёбра (корректность Dijkstra/A* сохраняется).

### 4.2 Стоимость и цель оптимизации

Цель — **лексикографическая тройка** на размещённых solver-ом клетках:
`(Σ scarcityPenalty, число клеток, число крафт-операций)` — сравнение покомпонентно, без «магических»
весов. Это убирает нулевые плато (даже изобильные клетки стоят `cellCost` во второй компоненте) и
делает решения минимальными по числу клеток при равном дефиците.

`scarcityPenalty` единицы аспекта = `obtainCost(X)` (рекурсивно, с учётом крафта):

```
obtainCost(X) = min( directPenalty(X),
                     obtainCost(c1) + obtainCost(c2) + ε )      // ε — мелкий оверхед крафта
directPenalty(X):
   supply[X] ≥ threshold     → 0                      // изобильный — нулевой штраф дефицита
   0 < supply[X] < threshold → base + k·(threshold - supply[X])   // дефицит штрафуется
   supply[X] == 0            → +∞                      // напрямую нельзя, только крафт
```

- ε обеспечивает: при нулевом прямом штрафе не крафтим зря; из равных крафтов предпочитаем мелкий.
- Праймалы не крафтятся ⇒ их `obtainCost = directPenalty`.
- `obtainCost` считается DP по DAG на стартовом инвентаре (статическая оценка дефицита для solver-а).
- Точные `base`, `k`, `ε`, `cellCost` фиксируются в коде и покрываются тестами на монотонность
  (изобильное << дефицитное; крафт из изобильного << прямой дефицит; меньше клеток предпочтительнее).

### 4.3 Аллокация, выполнимость и кнопка «Subtract used»

Статическая стоимость §4.2 **не гарантирует** агрегатную выполнимость по запасам (напр. 50 «бесплатных»
единиц, но размещений больше). Поэтому:

- **Проверка выполнимости перед принятием решения.** Для кандидата-решения выполняется аллокация
  ресурсов (направленный **DP/ограниченный бэктрекинг**, не жадность): для всего мультимножества
  размещённых аспектов ищется распределение «прямое vs крафт», умещающееся в `supply`. Порядок
  обработки детерминирован. Если выполнимой аллокации нет — кандидат **отвергается** (ветка
  отсекается); если ни одно валидное решение не выполнимо по запасам — статус `INFEASIBLE_INVENTORY`
  (решение по связям есть, но запасов не хватает).
- **Что списывается.** Заводится baseline-ревизия доски *до* Auto Solve (якоря + ручные locked).
  Списываются **только новые клетки**, размещённые solver-ом (`PLACED, locked=false`). Якоря и ранее
  введённые вручную аспекты **не** списываются (во избежание двойного учёта).
- **Subtract used.** Кнопка вычитает зафиксированное мультимножество листовых прямых расходов
  (с учётом выбранной аллокации крафта) из инвентаря; защита от ухода в минус; UI показывает разбивку
  «прямо потрачено / скрафчено».

Замечание: точный совместный оптимум распределения — это min-cost flow / ЛП; в v1 используется
DP/бэктрекинг-аллокация (корректная по выполнимости), точный поток — в §10.

## 5. Алгоритм solver-а

### 5.1 Постановка и границы

Node-weighted Steiner tree на произведении «доска × метки-аспекты» с ограничением полной валидности.
NP-трудно в общем случае; доски малы (≤91 клетка). Поддерживается **2–8 якорей** (тестируемый максимум);
поведение при **0/1 якоре** — тривиально «уже решено» (пустой набор размещений). Сверх максимума — UI
предупреждает и не запускает поиск.

### 5.2 Ядро (подход A): cost-bounded branch-and-bound (A*/IDA*)

- Состояние — частичное множество размещённых клеток (`coord → aspect`), стартующее от якорей +
  locked-клеток; цель — все якоря в одной связной компоненте.
- Расширение: добавить аспект в пустую клетку, соседнюю с текущей структурой. При каждом размещении
  проверяется валидность против **всех** уже стоящих соседей ⇒ полная валидность доски и обход dead hex.
- Стоимость — лексикографическая (§4.2); сравнение и отсечение по верхней границе — покомпонентно.
- **Допустимая эвристика h.** Используется оптимум *релаксации*: задача Штейнера на графе **клеток**
  (без меток и без полной валидности; вес клетки = мин. `obtainCost` достижимого там аспекта),
  решаемая **точно** алгоритмом **Dreyfus–Wagner** за O(3^k·n + 2^k·n²) (k = число несоединённых
  якорей, ≤8). Снятие ограничений ⇒ нижняя оценка ⇒ **допустимость** гарантирована. `h=0` —
  безопасный дефолт. **MST по попарным расстояниям ОТВЕРГНУТ** как переоценивающий (для общего
  центра вроде `air`, к которому примыкают `tree/aura/light`, MST даёт ~2w вместо w) — он неприемлем
  и не используется.
- **Anytime:** сначала быстрое допустимое решение (посев, §5.3), затем улучшение до оптимума или до
  бюджета; лучшее найденное — всегда доступно. Перед принятием — проверка выполнимости §4.3.

### 5.3 Посев (подход B): попарный Dijkstra + MST-сшивка — кандидаты

Для пар якорей — дешёвые пути в произведении-графе `(cell, aspect)`, метрическое замыкание, сшивка.
**Посев недоверенный:** результат Dijkstra сам по себе не гарантирует полную валидность (инцидентные
касания, совпадающие координаты с разными метками, примыкание третьего якоря, конфликты слияния).
Поэтому посев трактуется как **набор кандидатов**: слияние → полная `validate()` → при конфликте
отвергается/ремонтируется бэктрекингом. В дерево решения попадает только прошедший `validate()`.

### 5.4 Статусы результата и фолбэк

Результат solver-а имеет явный статус:
`OPTIMAL` | `FEASIBLE_TIMEOUT` | `INFEASIBLE_INVENTORY` | `UNSAT_PROVEN` | `CANCELLED` | `INVALID_INPUT`.

- `UNSAT_PROVEN` выдаётся **только** при исчерпывающем поиске (доказано отсутствие решения).
- **Beam-фолбэк** (ограниченная ширина) на тяжёлых досках даёт `FEASIBLE_TIMEOUT` или «решение не
  найдено (не доказано)», но **никогда** `UNSAT_PROVEN` (исчерпание beam не доказывает невыполнимость).
  Любое ограничение/усечение **явно логируется** (без «тихих» усечений).

### 5.5 Производительность, отмена, бюджеты

- Предвычисления: смежность клеток, битмаски доски, кэш эвристики, разложение аспектов.
- Порядок ветвления: сначала дешёвые/перспективные аспекты; отсечение по верхней границе.
- **Бюджеты по радиусам** (узлы/время/память) задаются явно для R2–R5 и проверяются бенчами (§7).
- **Web Worker.** Поиск идёт **чанками** с возвратом управления в асинхронный планировщик между
  чанками (проверка флага отмены/сообщений). **Гарантированная отмена — hard `worker.terminate()`**
  c пересозданием воркера (синхронный чанк не может «зависнуть» неотменяемо). Прогресс через
  `postMessage` (узлы / лучшая стоимость / время / статус).
- Решатель — **чистый TS без DOM** (тестируется напрямую `bun test` и используется воркером).

## 6. Архитектура

```
src/
  data/aspects.ts        # порт version_dictionary + addon_dictionary в TS; граф (Set); translate; иконки; стартовая валидация

  core/solver.ts         # B&B (A*/IDA*) + Dreyfus–Wagner h + beam-фолбэк; anytime; статусы; прогресс; отмена — чистый, без DOM
  worker/solver.worker.ts# обёртка воркера; запускается как new Worker(new URL('./solver.worker.ts', import.meta.url), {type:'module'})
  ui/...                 # рендер гекс-доски (SVG), палитры (Aspects + Inventory), инструменты, прогресс, атрибуция
  state/persistence.ts   # localStorage со schemaVersion + миграция/валидация
tests/                   # bun test: hex, граф, валидность, cost-модель, аллокация, корректность/статусы solver, бенчи
```

**Архитектурное требование:** воркер инстанцируется строго формой
`new Worker(new URL('./solver.worker.ts', import.meta.url), { type: 'module' })` — статически
анализируемой Vite для корректного переписывания base-пути. Каждый модуль — одна ответственность,
чистые интерфейсы, тестируется изолированно. Старый код не трогаем.

## 7. Тесты и производительность (test-first для ядра, ДО UI)

Порядок: сначала проходит тесты алгоритм, потом пишем UI. **Типизация:** `bunx tsc --noEmit` —
обязательный шаг (Vite/bun только транспилируют, не проверяют типы).

`bun test` покрывает:
- **hex.ts**: соседи, расстояния, генерация доски по радиусам, сериализация/десериализация.
- **aspectGraph.ts**: смежность из данных (эталонный edge-set фикстур), праймал-разложение, защита от
  циклов (fail loudly), `isValidLink` (запрет одинаковых соседей и «братьев»).
- **board.ts**: полная валидность (INVALID_LINK, SAME_ASPECT_ADJACENT, инцидентные касания,
  PLACED_ON_DEAD), связность якорей (ANCHORS_DISCONNECTED), isComplete; (де)сериализация с миграцией.
- **inventory.ts**: directPenalty по порогу; obtainCost с крафтом (изобильный компонент «спасает»
  дефицитный компаунд); ε-предпочтение мелких крафтов; монотонность; **аллокация-DP** (порядко-
  независимость, нахождение выполнимого распределения там, где жадность падает); корректное вычитание;
  отказ на отрицательных/нецелых входах.
- **solver.ts**: найденное решение **всегда** валидно и связно (инвариант); обход dead hex; мульти-
  якоря (до 8); **предвалидация входа** → `INVALID_INPUT`; заведомо неразрешимые доски →
  `UNSAT_PROVEN` только при исчерпывающем поиске; недостаток запасов → `INFEASIBLE_INVENTORY`;
  предпочтение изобильных аспектов; **оптимальность** на малых проверяемых кейсах; **допустимость
  эвристики** (h ≤ фактической стоимости на наборе кейсов; проверка, что A* с h не теряет оптимум).
- **Бенчи**: время/узлы по R2–R5 при типовых наборах якорей; фиксируем бюджеты §5.5 и применяем
  оптимизации. Бенч — отдельный скрипт, не блокирующий обычный прогон тестов.

## 8. Интерфейс (по референс-скриншоту)

- **Шапка:** заголовок; селектор радиуса (2–5); инструменты **Dead Hex**, **Erase**, **Clear**,
  **Auto Solve**; новые **Validate** и **Continue Solve**.
- **Лево — палитра `Aspects`:** клик по аспекту выбирает «кисть»; клик по гексу ставит якорь
  (или ручное размещение в режиме редактирования).
- **Центр — пергаментная гекс-доска (SVG):** иконки аспектов, dead hex заштрихованы; невалидные
  связи подсвечиваются красным; статус-строка снизу.
- **Право — `Inventory`:** список аспектов с редактируемыми количествами; поле порога «мало» (50);
  кнопка **Subtract used** + разбивка потраченного (прямо/крафт).
- **Прогресс:** во время решения — прогресс-бар (узлы / лучшая стоимость / время / статус) и
  кнопка **Cancel** (hard-terminate воркера).
- **Ручной режим:** клик по пустому гексу ставит выбранный аспект (locked); **Validate** проверяет
  полную валидность + связность и подсвечивает проблемы; **Continue Solve** фиксирует ручные
  размещения как ограничения и достраивает остальное.
- **Статусы решателя** отображаются человекочитаемо (Оптимально / По таймауту / Не хватает запасов /
  Решения нет / Отменено / Некорректный ввод).
- **Атрибуция:** видимая ссылка CC-BY-4.0 на исходные данные/иконки (по README репо) + уведомление
  в репозитории.
- Всё состояние (инвентарь, порог, доска, dead hex) — в **localStorage** со `schemaVersion`.

## 9. Деплой

- Рантайм/пакеты/тесты — **bun** (версия запинена); дев-сервер и бандл — **Vite** (TS, версия запинена);
  зависимости залочены (`bun.lockb`).
- **Обязательный шаг сборки:** `bunx tsc --noEmit` (типизация) перед `vite build`.
- Репозиторий — project page; GitHub Pages source = ветка `gh-pages`, **/(root)**. Новое приложение
  публикуется по фиксированному пути **`/v2/`** → доступно на `https://<user>.github.io/tcresearch-solver/v2/`.
  Соответственно Vite **`base = '/tcresearch-solver/v2/'`** (фиксируется в самом начале реализации),
  `build.outDir` = коммитимая папка `v2/` в корне ветки. **Старый сайт остаётся** по прежнему адресу
  (корень). Pipeline: `bun run build` → артефакты в `v2/` → коммит. Перед публикацией —
  **smoke-test прод-сборки именно под префиксом `/tcresearch-solver/v2/`** (JS, иконки, worker
  грузятся) и проверка, что корневой legacy-сайт продолжает открываться.

## 10. Возможные улучшения (вне первой версии)

- Точный совместный оптимум распределения ресурсов (min-cost flow / ЛП) вместо DP-аллокации §4.3.
- Замена набора данных на верифицированный полный датасет GTNH (архитектура §2.1 это допускает).
- Импорт доски из скриншота/буфера; пресеты исследований.
- Несколько решений на выбор (по дефициту / по числу клеток).
